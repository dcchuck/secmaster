# OTC/Microcap Secmaster — Product Design Spec

## Overview

A web application providing OTC and microcap security master data to retail researchers and small professional shops (RIAs, small quant funds). No clean, affordable product like this exists today.

The product is a SaaS web app (UI-first), with paid users able to provision API keys for programmatic access. The underlying data model is survivorship-safe, point-in-time-aware, and vendor-agnostic.

## Target Users

- **Retail/hobbyist traders** — microcap and OTC researchers who need real identity and status history
- **Small fund / professional** — RIAs, small quant shops, microcap fund managers who can't afford Bloomberg

## V1 Feature Scope

### In scope
- Security, issuer, and listing profile pages with full identity history
- OTC tier/status history, venue/trading visibility
- Corporate actions timeline (splits, mergers, symbol changes, delistings)
- Shares outstanding / float / market cap as-of history
- Multiple data ingestion sources (Fidelity existing work + more)
- Free/paid tier structure
- Paid users can provision API keys for programmatic access
- `as_of` date parameter for point-in-time views

### Phase 2 (out of v1 scope)
- Full-text search across issuers, securities, tickers
- Universe builder with filtering and export
- Per-user saved state (watchlists, saved screens)
- OAuth providers beyond what Clerk provides by default
- Billing integration (Stripe)

## Tech Stack

| Layer | Choice |
|-------|--------|
| Database | PostgreSQL |
| ORM + Schema | SQLModel (SQLAlchemy + Pydantic unified) |
| Migrations | Alembic (autogenerated from SQLModel models) |
| Backend | FastAPI (Python, strongly typed throughout) |
| Frontend | React + TypeScript + Vite |
| Auth | Clerk (OAuth, email/password, MFA — managed service) |
| API client generation | Auto-generated from FastAPI OpenAPI spec (orval or openapi-typescript-codegen) |
| Server state (frontend) | TanStack Query (React Query) |
| Deployment | Dockerized SaaS, cloud-hosted (Fly.io, Railway, or AWS) |

## Data Model

Adopted from the handoff document (`otc-secmaster-handoff.md`), implemented as SQLModel classes.

### Core tables

#### `issuer`
Company or legal entity. Fields: `issuer_id` (PK), `legal_name`, `normalized_name`, `issuer_type`, `country_incorporation`, `domicile_country`, `cik`, `lei`, `sic`, `naics`, `is_shell_flag`, `is_bankrupt_flag`, `is_liquidating_flag`, `created_at`, `updated_at`.

#### `issuer_name_history`
Historical issuer names. Fields: `id` (PK), `issuer_id` (FK), `name`, `normalized_name`, `effective_start_date`, `effective_end_date`, `source`, `created_at`.

#### `security`
Financial instrument linked to an issuer. Fields: `security_id` (PK), `issuer_id` (FK), `security_type`, `security_subtype`, `share_class`, `par_value`, `currency`, `is_primary_equity_flag`, `underlying_security_id` (self-FK), `created_at`, `updated_at`.

#### `security_identifier_history`
Historical identifiers (ticker, CUSIP, ISIN, FIGI, etc.). Fields: `id` (PK), `security_id` (FK), `id_type`, `id_value`, `venue_code`, `effective_start_date`, `effective_end_date`, `is_primary_flag`, `source`, `created_at`.

#### `listing`
Security on a venue over time. Fields: `listing_id` (PK), `security_id` (FK), `venue_code`, `mic_code`, `primary_symbol`, `currency`, `country`, `listing_status`, `effective_start_date`, `effective_end_date`, `is_primary_listing_flag`, `created_at`, `updated_at`.

#### `listing_status_history`
OTC tier, compliance, and tradability context over time. Fields: `id` (PK), `listing_id` (FK), `status_date`, `listing_status`, `tier`, `caveat_emptor_flag`, `unsolicited_quotes_only_flag`, `shell_risk_flag`, `sec_suspension_flag`, `bankruptcy_flag`, `current_information_flag`, `transfer_agent_verified_flag`, `source`, `created_at`.

#### `corporate_action`
Lifecycle and continuity-changing events. Fields: `corporate_action_id` (PK), `security_id` (FK), `issuer_id` (FK), `action_type`, `announcement_date`, `effective_date`, `ratio_from`, `ratio_to`, `old_value`, `new_value`, `notes`, `source`, `created_at`.

#### `shares_outstanding_history`
Point-in-time capital structure. Fields: `id` (PK), `security_id` (FK), `as_of_date`, `shares_outstanding`, `public_float`, `authorized_shares`, `market_cap`, `enterprise_value`, `value_source_type`, `source`, `created_at`.

#### `vendor_security_map`
Maps vendor identities to canonical internal entities. Fields: `id` (PK), `vendor_name`, `vendor_entity_type`, `vendor_id`, `issuer_id` (FK), `security_id` (FK), `listing_id` (FK), `effective_start_date`, `effective_end_date`, `confidence_score`, `mapping_method`, `created_at`.

### Product tables

#### `user`
Application user linked to Clerk. Fields: `user_id` (PK), `clerk_user_id` (unique, indexed), `email`, `tier` (free/paid), `created_at`, `updated_at`.

#### `api_key`
API keys provisioned by paid users. Fields: `id` (PK), `user_id` (FK), `key_hash`, `label`, `created_at`, `last_used_at`, `is_active`.

### Data model principles
- Internal canonical IDs (`issuer_id`, `security_id`, `listing_id`) — never ticker as PK
- `effective_start_date` / `effective_end_date` on all history tables
- `source` field on all ingested data for vendor lineage
- `created_at` / `updated_at` on all tables
- Categorical fields (OTC tier, listing status, action type) are string columns with Python-side enum validation — no Postgres enums

### Point-in-time query pattern
```sql
WHERE effective_start_date <= :as_of_date
  AND (effective_end_date IS NULL OR effective_end_date > :as_of_date)
```

## API Design

All routes under `/api/v1/`. Versioned from the start since paid users depend on API stability.

### Endpoints

| Area | Endpoints | Description |
|------|-----------|-------------|
| Issuers | `GET /issuers`, `GET /issuers/{id}`, `GET /issuers/{id}/history` | Browse + issuer profile with name/classification history |
| Securities | `GET /securities`, `GET /securities/{id}`, `GET /securities/{id}/identifiers`, `GET /securities/{id}/actions` | Browse + security detail, identifier history, corporate actions |
| Listings | `GET /listings`, `GET /listings/{id}`, `GET /listings/{id}/status-history` | Browse + listing detail, OTC tier/status timeline |
| API Keys | `POST /api-keys`, `GET /api-keys`, `DELETE /api-keys/{id}` | Paid users provision/manage keys |

### Query parameters
- `?as_of=YYYY-MM-DD` — optional on all read endpoints; when present, all history joins use effective dating filter; when absent, returns current state
- Cursor-based pagination on all list endpoints

## Authentication

**Provider:** Clerk (managed auth service).

- **Frontend:** Clerk React SDK handles login/signup UI, OAuth flows, session management
- **Backend:** Clerk JWT verification middleware on FastAPI validates tokens on every request
- **User sync:** On first authenticated request, a `user` record is created with `clerk_user_id` link
- **API keys:** Managed by the app. Paid users provision keys stored in the DB. Separate middleware path resolves API key to internal user.
- **No auth routes in the API** — Clerk handles login, register, OAuth callbacks, password reset, MFA

### Tier enforcement
- **Free:** Browse data, view profiles, rate-limited
- **Paid:** Higher rate limits, API key provisioning, export
- Enforced via middleware reading `user.tier`
- No billing integration for v1 (manual upgrades; Stripe later)

## Frontend Architecture

- **Framework:** React + TypeScript + Vite
- **API client:** Auto-generated from FastAPI's OpenAPI spec
- **Server state:** TanStack Query (React Query) — caching, pagination, refetching
- **Routing:** React Router — flat routes (`/issuers/:id`, `/securities/:id`, `/listings/:id`)
- **Design:** Desktop-first, responsive. This is a research tool.

### Pages (v1)
- Login / Register (Clerk-managed)
- Issuers list — paginated browse, basic column filtering
- Issuer profile — name history timeline, linked securities, classification history, flags
- Security profile — identifier history, linked listings, corporate actions timeline, shares outstanding
- Listing profile — venue info, OTC tier/status history, caveat emptor and other flags over time
- API key management — generate/revoke keys, view usage (paid tier only)

### Core UI patterns
- History tables and timelines are the primary component — effective-dated records shown chronologically
- Profile pages link between entities (issuer → securities → listings)
- `as_of` date picker on profile pages for point-in-time state

## Ingestion Architecture

Each vendor implements a common interface:

```python
class BaseIngestor(ABC):
    @abstractmethod
    async def fetch(self) -> list[RawRecord]: ...

    @abstractmethod
    async def transform(self, raw: list[RawRecord]) -> list[SQLModel]: ...

    async def load(self, records: list[SQLModel]) -> IngestResult: ...
```

- **`fetch`** — pull raw data from vendor (API, file, scrape)
- **`transform`** — normalize into canonical SQLModel instances
- **`load`** — upsert into Postgres (shared base class implementation)

### Running ingestion
- CLI: `python -m app.ingestion.cli fidelity`
- Can be triggered as FastAPI background tasks for admin-initiated refreshes
- No separate worker infrastructure for v1 — CLI commands run manually or via cron

### Vendor mapping
Each ingestor writes to `vendor_security_map` during load, linking vendor IDs to canonical IDs. Matching logic lives in `app/services/identity.py`.

### Idempotency
Ingestion runs are safe to re-run. Upserts keyed on vendor + vendor_id + effective_date.

## Deployment

### Local dev
- `docker-compose.yml`: Postgres + FastAPI app
- Vite dev server with proxy to API
- Alembic migrations run locally

### Production
- Dockerized FastAPI (uvicorn) + static frontend on CDN
- Single Postgres instance
- Target: Fly.io, Railway, or AWS ECS
- `alembic upgrade head` runs as part of deploy pipeline

### Configuration
- `pydantic-settings` reading from env vars / `.env`
- Database URL, Clerk keys, rate limit config, etc.
